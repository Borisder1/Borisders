# Огляд проєкту Borisders

## 1) Поточний стан

На момент огляду репозиторій містив лише один файл із набором команд для ручної ініціалізації git.

### Ризики такого стану

- Немає опису цілей проєкту (важко для онбордингу).
- Немає відтворюваного процесу запуску/збірки.
- Немає базової інженерної інфраструктури (CI, тестів, шаблонів).

## 2) Що покращено

- Перенесено одноразові команди у виконуваний скрипт `scripts/bootstrap_repo.sh`.
- Додано базову документацію (`README.md`) для старту.

## 3) Рекомендації (пріоритети)

### P0 — зробити відразу

1. Визначити призначення проєкту (1-2 абзаци + цільова аудиторія).
2. Затвердити стек (наприклад: Node.js / Python / Go).
3. Додати мінімальний CI-процес:
   - перевірка форматування;
   - запуск тестів;
   - перевірка збірки.

### P1 — найближчий спринт

1. Структура директорій:
   - `src/` — код;
   - `tests/` — тести;
   - `docs/` — технічна документація.
2. Шаблони GitHub:
   - `.github/PULL_REQUEST_TEMPLATE.md`;
   - `.github/ISSUE_TEMPLATE/`.
3. Ліцензія (`LICENSE`) і `CONTRIBUTING.md`.

### P2 — масштабування

1. Автоматизація релізів (семантичні теги, changelog).
2. Метрики якості (coverage, lints).
3. Базова безпека:
   - `dependabot`;
   - секрет-сканування;
   - policy для protected branches.

## 4) Технічна думка

Проєкт у правильному місці для старту, але зараз це радше "заготовка" репозиторію, а не продукт. Найкращий наступний крок — зафіксувати продуктову ідею та обрати базовий стек, після чого швидко підняти skeleton застосунку з CI.
